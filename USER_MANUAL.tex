\section{Monorepo Architecture \& Services}
The HPE SAN Unified Platform unifies a virtualized storage network, an automated crawler engine, a graph-based topology viewer, and cutting-edge artificial intelligence agents to streamline SAN system operations.

\subsection{Core Services \& Directory Structure}
The system comprises five core services operating in unison:
\begin{enumerate}
    \item \textbf{React Vite Frontend Dashboard} (Port \texttt{3000}): Interactive UI using React Flow, Cytoscape visualizations, resizable column grids, live-typing terminal shells, and WebSocket/SSE log streaming.
    \item \textbf{Flask Master REST API} (Port \texttt{5005}): Orchestration layer serving Neo4j data transformations, synthetic database models, Elasticsearch routing, and RAG agent logic.
    \item \textbf{Python Network Simulator} (Port \texttt{5001}): Replays production CLI commands, models multipath arrays, and manages an in-process socket routing table.
    \item \textbf{Node.js + Express Chatbot Service} (Port \texttt{5010}): Manages user authentication, multi-tenant RBAC permissions, and persistent MongoDB chat history.
    \item \textbf{Database Cluster}:
    \begin{itemize}
        \item \textbf{Neo4j} (Port \texttt{7474}): Maps physical/logical relationships (e.g., switches connected to storage node ports).
        \item \textbf{Elasticsearch} (Port \texttt{9200}): Drives the indexer engine for high-speed cross-component string searches.
        \item \textbf{MongoDB} (Port \texttt{27017}): Retains application schemas, user credentials, and historical LLM chat records.
    \end{itemize}
\end{enumerate}

\subsection{Ontological Data Schema}
The SAN is modeled as a directed graph representing real-world SAN physical dependencies. The relationships are illustrated in the following structural chain:
\begin{equation*}
\text{ArraySystem} \xrightarrow{\text{HAS\_NODE}} \text{Node/Controller} \xrightarrow{\text{HAS\_PORT}} \text{Port} \xrightarrow{\text{CONNECTS\_TO}} \text{Host HBA} \xrightarrow{\text{CONNECTS\_TO}} \text{ArraySystem}
\end{equation*}
Additionally, array enclosures are modeled hierarchically:
\begin{equation*}
\text{ArraySystem} \xrightarrow{\text{HAS\_CAGE}} \text{Cage} \xrightarrow{\text{CONTAINS}} \text{PhysicalDisk}
\end{equation*}

\subsection{Lightweight OS Device Fingerprinting}
During discovery crawl runs, the system classification script identifies the vendor hardware before executing queries:
\begin{lstlisting}[language=Python, caption=Fingerprint Classification Pipeline]
def fingerprint_device(ip: str, virtual_network) -> DeviceType:
    meta = virtual_network.get_metadata(ip)
    if meta.get("type") == "array":
        return DeviceType.HPE_ARRAY
    if meta.get("type") == "switch":
        return DeviceType.HPE_ARRAY
    if meta.get("type") == "host":
        if meta.get("os_type") == "windows":
            return DeviceType.WINDOWS
        return DeviceType.LINUX

    # Active Shell Probing Fallback
    probe = virtual_network.execute(ip, "showsys")
    if probe and "Connection refused" not in probe:
        if "TotalCap" in probe or "AllocCap" in probe:
            return DeviceType.HPE_ARRAY
            
    probe = virtual_network.execute(ip, "uname -a")
    if probe and "Linux" in probe:
        return DeviceType.LINUX
        
    return DeviceType.UNKNOWN
\end{lstlisting}

\subsection{The Iterative BFS Network Discovery Crawler}
When a user launches the discovery engine, the crawler utilizes an iterative BFS traversal pattern:
\begin{lstlisting}[language=Python, caption=Discovery BFS Core Pipeline]
class DiscoveryCrawler:
    def discover(self, seed_ips: List[str]):
        self.running = True
        self.visited = set()
        self.queue = deque(seed_ips)
        
        while self.queue:
            ip = self.queue.popleft()
            if ip in self.visited:
                continue
            self.visited.add(ip)
            
            # Fingerprint and execute commands
            dev_type = fingerprint_device(ip)
            if dev_type == "hpe_array":
                outputs = self._run_commands(ip, HPE_COMMANDS)
                parsed = parse_sim_array_output(outputs)
                # Store in databases
                self.neo4j.store(parsed)
                self.es.index(parsed)
                self.mongo.store(parsed)
                # Extract and enqueue newly found IPs
                new_ips = self._extract_linked_ips(parsed)
                for new_ip in new_ips:
                    if new_ip not in self.visited:
                        self.queue.append(new_ip)
\end{lstlisting}

\subsection{LLM Diagnostic Logic \& The Stateful SAN Agent}
When executing the \textbf{SAN Agent} mode in the AI Assistant, the platform deploys a stateful reasoning loop:
\begin{enumerate}
    \item \textbf{Plan Formulation}: The LLM evaluates the user's natural language query and determines which CLI commands (e.g., \texttt{showsys}, \texttt{showhost}) need to run on the target IP array.
    \item \textbf{Tool Execution}: The agent connects to the Simulator's CLI engine and executes the planned commands.
    \item \textbf{Real-Time Parsing}: Outputs are run through specialized regex parsers to extract active entities (e.g., degraded disks, host WWNs).
    \item \textbf{Graph Enrichment}: The parsed entities are upserted into Neo4j to ensure the topology graph is in sync with the live device state.
    \item \textbf{Answer Formulation}: The LLM synthesizes the diagnostic timeline and the final answer, streaming the entire trace back to the user.
\end{enumerate}
